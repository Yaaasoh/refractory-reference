# Consistency Checker

**Version**: 1.0.0
**カテゴリ**: 分析フレームワーク
**出典**: critical-review-frameworks.md

## 概要

整合性チェックと完全性チェックを統合したスキル。
以下の2つのフレームワークを提供:

1. **整合性チェック**: 複数の情報源・要素間の矛盾を検出
2. **完全性チェック**: 必要な要素がすべて含まれているかを確認

---

## フレームワーク1: 整合性チェック

### 目的

複数の情報源、ドキュメント、コード間の矛盾や不整合を検出する。

### 実行プロセス

```
Step 1: チェック対象の特定
    ↓
Step 2: 比較項目の抽出
    ↓
Step 3: クロスリファレンス検証
    ↓
Step 4: 不整合の記録
    ↓
Step 5: 修正提案
```

### チェック観点

| 観点 | 説明 | 例 |
|------|------|-----|
| **名称の一致** | 同じものに同じ名前を使用 | 関数名、変数名、用語 |
| **数値の一致** | カウント、統計が一致 | ファイル数、行数、件数 |
| **日付の整合性** | 時系列に矛盾なし | 作成日 < 更新日 |
| **参照の有効性** | リンク、参照が有効 | URL、ファイルパス、関数参照 |
| **構造の一貫性** | 同じパターンの適用 | 命名規則、フォーマット |
| **仕様と実装** | 仕様書とコードの一致 | API定義、データ構造 |

### 出力テンプレート

```markdown
## 整合性チェック結果

### チェック対象
| # | 対象 | 種類 | 場所 |
|:-:|------|------|------|
| 1 | [対象1] | [種類] | [ファイルパス等] |
| 2 | [対象2] | [種類] | [ファイルパス等] |

### チェック項目
| # | 項目A | 項目B | 整合性 | 備考 |
|:-:|-------|-------|:------:|------|
| 1 | [参照元の内容] | [参照先の内容] | ✓/✗ | [詳細] |
| 2 | ... | ... | ... | ... |

### 検出された不整合

#### 不整合 #1: [タイトル]
- **場所A**: [ファイル:行番号] - 「[内容A]」
- **場所B**: [ファイル:行番号] - 「[内容B]」
- **矛盾内容**: [何が矛盾しているか]
- **重大度**: 高/中/低
- **修正提案**: [どう修正すべきか]

#### 不整合 #2: ...

### サマリ
| 項目 | 値 |
|------|-----|
| チェック項目数 | X |
| 整合 | Y件 |
| 不整合 | Z件 |
| 整合率 | XX% |

### 推奨アクション
1. [優先度1のアクション]
2. [優先度2のアクション]
```

### チェックリスト

- [ ] すべての参照元・参照先を特定したか？
- [ ] 同じ情報が複数箇所にある場合、すべて確認したか？
- [ ] 不整合の重大度を判定したか？
- [ ] 修正提案は具体的か？

## フレームワーク2: 完全性チェック

### 目的

必要な要素がすべて含まれているか、欠落がないかを確認する。

### 実行プロセス

```
Step 1: 必須項目リストの定義
    ↓
Step 2: 対象の読み込み
    ↓
Step 3: 各項目の存在確認
    ↓
Step 4: 品質評価
    ↓
Step 5: 欠落項目の報告
```

### 共通チェック項目（ドキュメント向け）

| カテゴリ | チェック項目 |
|---------|------------|
| **基本情報** | タイトル、作成日、作成者、バージョン |
| **目的・概要** | 背景、目的、スコープ |
| **本体** | 主要コンテンツ、詳細説明 |
| **参照** | 前提条件、関連ドキュメント、用語定義 |
| **メタ情報** | 改訂履歴、承認情報 |

### 出力テンプレート

```markdown
## 完全性チェック結果

### チェック対象
- **対象**: [ドキュメント名/コード名]
- **種類**: [計画書/仕様書/コード等]
- **チェック基準**: [使用した基準]

### 必須項目の存在確認
| # | 必須項目 | 存在 | 品質 | 備考 |
|:-:|---------|:----:|:----:|------|
| 1 | [項目名] | ✓/✗ | A/B/C/D | [不足内容] |
| 2 | [項目名] | ✓/✗ | A/B/C/D | [不足内容] |
| ... | ... | ... | ... | ... |

### 品質グレード定義
| グレード | 説明 |
|:--------:|------|
| A | 十分な記述（詳細かつ明確） |
| B | 記述あり（改善余地あり） |
| C | 最小限の記述 |
| D | 不十分（要補完） |
| - | 未記載 |

### サマリ
| 項目 | 値 |
|------|-----|
| 必須項目数 | X |
| 存在 | Y件 |
| 欠落 | Z件 |
| **充足率** | **XX%** |
| 品質A | W件 |
| 品質B-D | V件 |

| # | 欠落項目 | 重要度 | 補完提案 |
|:-:|---------|:------:|---------|
| 1 | [項目名] | 高/中/低 | [何を追記すべきか] |

### 品質懸念項目（推奨対応）
| # | 項目 | 現状品質 | 改善提案 |
|:-:|------|:--------:|---------|

**[完全/ほぼ完全/要補完/不完全]**

[総合的なコメント]
```

### 用途別チェックリスト

#### 計画書向け
- [ ] 目的・背景が明確か
- [ ] スコープ（対象・対象外）が定義されているか
- [ ] 成功基準が測定可能か
- [ ] リスクと対策が含まれているか
- [ ] スケジュール/マイルストーンがあるか

#### 仕様書向け
- [ ] 機能要件が網羅されているか
- [ ] 非機能要件（性能、セキュリティ等）があるか
- [ ] インターフェース定義があるか
- [ ] エラーケースが考慮されているか
- [ ] 用語定義があるか

#### コード向け
- [ ] 必須関数/クラスが実装されているか
- [ ] エラーハンドリングがあるか
- [ ] テストがあるか
- [ ] ドキュメント/コメントがあるか
- [ ] 設定ファイルが揃っているか

## 使用場面

| 場面 | 使用フレームワーク | 例 |
|------|-------------------|-----|
| 「矛盾がないかチェックして」 | 整合性チェック | 仕様書と実装の照合 |
| 「漏れがないか確認して」 | 完全性チェック | 計画書のレビュー |
| 「一貫性を確認して」 | 整合性チェック | ドキュメント間の用語統一 |
| 「必要な項目が揃っているか」 | 完全性チェック | チェックリスト照合 |
| 「抜け漏れを指摘して」 | 完全性チェック | 要件の網羅性確認 |

---

## フレームワーク選択ガイド

```
複数の情報源の「矛盾」を探す → 整合性チェック
単一対象の「漏れ」を探す → 完全性チェック
両方必要な場合 → 完全性チェック → 整合性チェック
```

---

## 品質基準

### 整合性チェック
- チェック対象が明確に特定されているか
- 不整合は具体的な箇所（ファイル:行番号等）で示されているか
- 不整合の重大度が判定されているか
- 修正提案が具体的か

### 完全性チェック
- 必須項目リストは適切か（過不足なし）
- 各項目の存在だけでなく品質も評価しているか
- 欠落項目に対する補完提案があるか
- 総合評価の根拠が明確か

---

## アンチパターン

| アンチパターン | 問題 | 正しいアプローチ |
|---------------|------|-----------------|
| 「矛盾なし」で終了 | 確認範囲が不明 | チェック項目を明示 |
| 存在確認のみ | 品質が不明 | 品質グレードも評価 |
| 汎用チェックリスト依存 | 対象に合わない | 対象に応じてカスタマイズ |
| 重大度なしの列挙 | 優先順位不明 | 重大度を判定 |
