## 段階的詳細化ワークフロー

### Phase 1: 構造（スケルトン）

要素と接続関係のみを定義する。位置・色は一切指定しない。

```
以下の要素を含むブロック図のスケルトンを作成:
- コンポーネントA, B, C, D
- A->B, B->C, B->D の接続
位置とサイズのみ、スタイリングは後で。
```

**チェック**: 要素数が9以下か？ 超えるなら分割（後述）。

### Phase 2: レイアウト調整

配置・間隔・整列を決める。

```
前のSVGのレイアウトを調整:
- AとBを上段に横並び
- CとDを下段に横並び
- 要素間の間隔を均等に（グリッド100px刻み）
- 同じ階層のノードはy座標を揃える
```

### Phase 3: スタイリング

カラーパレット・フォント・線種を適用する。

```
以下のカラーパレットを適用:
- ノード背景: #2C2C2E
- テキスト: #E0E0E0
- 矢印: #78A9FF
- ボーダー: #444444
角丸8px、フォントNoto Sans JP、線の太さ1.5px
```

### Phase 4: 微調整

テキスト折り返し・ラベル修正・タイトル追加。

```
以下を修正:
- Bノードのテキストを2行に分割
- C->Dに破線の矢印を追加
- 全体にタイトル「システム構成図」を追加
```

## 制約テンプレート

図の生成コードを書く前に、以下の制約を必ず定義する:

```
【描画制約】
- フォーマット: [SVG / HTML+CSS / Mermaid / Graphviz]
- 描画領域: viewBox="0 0 800 600"（またはコンテナ幅100%）
- グリッド: 100px刻み
- 最大要素数: [N] (9以下)
- フォント: "Noto Sans JP", sans-serif
- フォントサイズ: 最小12px、最大24px

【カラーパレット】（5色以内で定義）
- 背景: [色コード]
- テキスト: [色コード]
- ノード背景: [色コード]
- アクセント: [色コード]
- ボーダー/矢印: [色コード]

【レイアウト】
- 方向: [左→右 / 上→下]
- ノード間最小距離: 40px
- 角丸半径: 8px
- 線の太さ: 1.5px
```

## 図種×手法の選択マトリクス

### 判断フロー

```
CJK長文ラベルか？
  Yes → HTML+CSS div（第1推奨）
  No → ラベルは短い（1-3語）か？
    Yes → Mermaid flowchart
    No → 幾何学的形状（ピラミッド等）か？
      Yes → インラインSVG（テンプレート編集）
      No → HTML+CSS div
英語ラベルのグラフ構造か？
  Yes → Graphviz（DOT）
  No → HTML+CSS div
```

### 詳細マトリクス

| 図種 | 第1推奨 | 第2推奨 | 理由 |
|:---|:---|:---|:---|
| フローチャート（短ラベル） | Mermaid | HTML+CSS | Mermaidの記述簡潔性 |
| CJK長文ラベルの図 | HTML+CSS | インラインSVG | テキスト折り返しが確実 |
| 判断分岐フロー | HTML+CSS | Mermaid | 菱形表現+ダーク対応 |
| 規格ピラミッド | HTML+CSS | インラインSVG | 幅変動レイアウト |
| ボウタイ図/対称構造 | HTML+CSS (Grid) | インラインSVG | 左右対称表現 |
| グラフ構造（英語） | Graphviz | Mermaid | 自動レイアウト |

## Few-Shot プロンプト例

### 良い仕様の書き方

**例1: システム構成図**
```
以下の要件でHTML+CSSダイアグラムを作成してください:

【内容】厳密に5個の要素のみ:
1. フロントエンド（React）
2. API Gateway
3. 認証サービス
4. ビジネスロジック
5. PostgreSQL

【接続】: 1->2, 2->3, 2->4, 4->5

【レイアウト】: 左から右、3段構成
- 左: フロントエンド
- 中央: API Gateway（上）、認証（下）
- 右: ビジネスロジック、PostgreSQL

【スタイル】:
- 配色: 抑制的で洗練された（restrained and sophisticated）
- 色数: 背景#1E1E1E、テキスト#E0E0E0、ノード#2C2C2E、アクセント#78A9FF、線#666
- フォント: "Noto Sans JP", sans-serif、14px
- 角丸: 8px、線幅1.5px
- 装飾なし（影なし、グラデーションなし）
```

**例2: プロセスフロー**
```
以下のプロセスフローをMermaidで作成してください:

【フロー方向】: 左から右（LR）
【ノード数】: 厳密に4個
  A[Start] --> B[Process] --> C{Decision}
  C -->|Yes| D[End]
  C -->|No| B

【制約】:
- 追加のノードやサブグラフは不要
- 色指定なし（デフォルト）
```

## 失敗パターンと回避策

### 1. 座標計算ミス

**症状**: SVGの要素が意図しない位置に配置される
**原因**: LLMの空間的推論の限界
**回避策**:
- 座標を100刻みグリッドで指定（x=100, 200, 300...）
- 相対配置（「前の要素の右に150px」）で指示
- 可能ならHTML+CSS（flexbox/grid）でブラウザに計算させる

### 2. 要素の重なり

**症状**: ノードやラベルが重なり合う
**原因**: 5ノード以上で空間配分が破綻
**回避策**:
- 1図あたり7-9要素以下に制限
- 超える場合は分割（後述の分割戦略）
- Mermaid/Graphvizのレイアウトエンジンに委ねる

### 3. テキストはみ出し（特にCJK）

**症状**: 日本語テキストがノードからはみ出す
**原因**: CJK文字幅の計算不正確
**回避策**:
- HTML+CSS: `word-break: break-all` + `overflow-wrap: break-word`
- SVG: `foreignObject` + CSSテキスト折り返し
- Mermaid: Markdown strings（`` ["`text`"] ``）で自動折り返し
- ラベル最大文字数を指定（例: 1行10文字以内）

### 4. 色ドリフト

**症状**: 修正を繰り返すうちにカラーパレットが崩壊
**原因**: 部分的な色変更で全体の一貫性喪失
**回避策**:
- カラーパレットをCSS変数で定義（`:root { --primary: #78A9FF; }`）
- 修正時「カラーパレットを維持」と必ず明示
- 色コードを直書きせず変数参照（`var(--primary)`）

### 5. 過剰な要素生成

**症状**: 指示した以上のノード・矢印・テキストが追加される
**原因**: LLMが「親切心」で情報を補完
**回避策**:
- 「厳密に以下のN個の要素のみ描画。追加要素は不要。」と明示
- 要素リストを番号付きで列挙

### 6. 構文エラーのループ修正

**症状**: Mermaid/SVGの構文エラー修正が新エラーを生む無限ループ
**原因**: 修正文脈の不足
**回避策**:
- エラーメッセージ全文をフィードバック
- 3回修正しても直らなければ最初から書き直し
- Mermaid: 特殊文字（`(`, `)`, `[`, `]`）はクォートで囲む

### 7. ドメイン固有記法の不正確

**症状**: UML、BPMN等の記号・表記法が正しくない
**回避策**:
- 記法のルールをプロンプトに含める
- テンプレートを提供して編集させる
- 厳密さが必要ならMermaid等の構造化記法を使用

### 8. セッション間の不一致

**症状**: 同じプロンプトでも生成結果が異なる
**回避策**:
- テンプレート編集方式で変動を最小化
- 生成結果をバージョン管理（Git）

## 検証ワークフロー

図の作成・修正後、以下の順序で検証する:

### 必須検証ステップ

1. **保存** → `diagram_quality_check.sh`がL1構文+L2品質を自動チェック（PostToolUse）
2. **警告確認** → L1/L2警告が出た場合は修正してから次へ
3. **レンダリング** → ブラウザでプレビュー（`quarto preview`、HTMLファイルを直接開く等）
4. **スクリーンショット** → 表示結果を画像として保存（証拠として必要）
5. **diagram-verifier** → サブエージェントでCRAP原則+技術品質を網羅チェック（推奨）
6. **完了報告** → 視覚検証の証拠を含めて報告（`prevent_false_completion.sh`が証拠を要求）

### diagram-verifierサブエージェントの呼び出し例

```
Agent(subagent_type="diagram-verifier", prompt="以下の図ファイルの品質をレビューしてください: path/to/diagram.qmd")
```

### facility-safety固有ツール

facility-safetyリポジトリでは追加の検証ツールが利用可能:
- `scripts/validate_diagrams.py` — Mermaid/SVG構文検証
- `scripts/verify_diagram_render.sh` — Quarto+Chromiumレンダリング検証
- `scripts/visual_verify_diagrams.mjs` — Playwright視覚テスト
- `scripts/visual_verify_site.mjs` — サイト全体の視覚テスト

## デザインレビューチェックリスト

### CRAP原則チェック

生成した図に対して以下を確認:

- [ ] **Contrast（対比）**: 重要要素は大きく・濃い。些細な違いではなく大胆な差
- [ ] **Repetition（反復）**: 同レベル要素は同じスタイル（色・形・線種が統一）
- [ ] **Alignment（整列）**: 全要素がグリッドに揃っている。上端・中心の整列
- [ ] **Proximity（近接）**: 関連要素は近く、無関連要素は離れている。余白でグループ表現

### ゲシュタルト法則チェック

- [ ] **近接**: 同カテゴリの要素が密集配置されているか
- [ ] **類同**: 同種の要素は同じ色・形状か
- [ ] **連続**: フロー線は交差を避け滑らかか
- [ ] **接続**: 関連要素は線で明示的に接続されているか
- [ ] **包囲**: グループは背景色や枠線で囲まれているか
- [ ] **矛盾なし**: 近接では同グループだが色が異なる等の矛盾がないか

### 技術品質チェック

- [ ] 要素数が9以下か（超えているなら分割を検討）
- [ ] テキストがノードからはみ出していないか
- [ ] カラーパレットが5色以内で一貫しているか
- [ ] コントラスト比がWCAG AA基準を満たすか（テキスト4.5:1、非テキスト3:1）
- [ ] 色以外の手段（形状・線種・ラベル）でも情報が区別できるか
- [ ] 装飾が最小限か（影なし、グラデーションなし、3D効果なし）

## 分割戦略

要素数が9を超える場合、以下の戦略で図を分割する:

### 1. 階層分割

- **Level 0**: システム全体の概要図（5要素以内）
- **Level 1**: サブシステムごとの詳細図
- **Level 2**: コンポーネントの内部構造図
- 各レベルで要素数を7-9以内に維持

### 2. レイヤー分割

- 物理構成図と論理構成図を別図にする
- 「関心事の分離」の原則を図にも適用

### 3. 段階的生成（大規模図の場合）

1. スケルトン生成（要素の配置のみ）
2. 接続関係の追加
3. ラベル・テキストの追加
4. スタイリング
5. 微調整

## テンプレート編集方式の手順

> 「テンプレートを生成するのではなく、テンプレートを編集する」
> -- PPTAgent, EMNLP 2025

### 手順

1. プロジェクト内のテンプレートディレクトリを確認（`styles/templates/` 等）
2. 図種に適したテンプレートを選択（なければ類似の既存図をベースに）
3. テンプレートをコピー
4. **テキスト内容のみ**を書き換え
5. レイアウト・色・フォントはテンプレートの定義を維持
6. ビルドして確認

### 分業設計

| 工程 | 担当 | 理由 |
|:---|:---|:---|
| テンプレート選択 | Claude Code | カタログ参照 |
| テキスト埋め込み | Claude Code | 得意領域 |
| ビルド確認 | ビルドツール | 自動検証 |
| 目視確認（ダーク/ライト） | 人間 | 視覚品質は主観 |
| 微調整（配置・余白） | 人間 | 座標調整はAI不安定 |

## 視覚的階層の設計

効果の強い順:

1. **サイズ**: 大きい要素が最も注目される
2. **色/コントラスト**: 高コントラストが注目を集める
3. **位置**: 上・左が先に読まれる
4. **余白**: 周囲の余白が多い要素が際立つ
5. **太さ**: 太い線・太字が強調される
6. **形状**: 異なる形状が注意を引く

### 技術図での階層設計例

- レベル1（最重要）: 大きいノード、濃い色、太い枠
- レベル2: 中サイズ、中間色
- レベル3: 小さいノード、淡い色、細い枠
- 接続線: ノードより控えめ（灰色、細め）
