# 文字起こし修正スキル（サンプル）

会議文字起こし修正のための構造化ワークフロー。他のドメイン固有ワークフロー作成の参考になる。

## このスキルの特徴

### 構造化されたPhase 0-4アプローチ

| Phase | 名称 | 目的 | 所要時間目安 |
|-------|------|------|-------------|
| 0 | 計画・分割 | 大規模ファイルの分割計画 | 15分 |
| 1 | 準備 | 用語集・参照資料の確認 | 30分 |
| 2 | メイン作業 | 誤変換修正・話者特定 | 2-3時間 |
| 3 | 二次成果物 | 議事メモ作成 | 1-2時間 |
| 4 | レビュー・プッシュ | 検証・Git操作 | 30分 |

### 核心原則

1. **ファイル分割必須**: 400行以上は必ず分割（Claude能力限界対策）
2. **用語集参照必須**: 推測ではなく用語集に基づく修正
3. **検証プロセス必須**: 1行1行確認、grep検証
4. **生データ保護**: 原本は編集禁止、修正版は別ファイル

## 適用時の判断基準

### このスキルを適用する場合

- 会議文字起こし（Whisper等）の修正
- OCR出力（古い技術文献等）の修正
- 話者特定・見出し追加が必要な場合
- 専門用語の誤変換修正が必要な場合

### 適用しない場合

- 要約・サマリ作成（文字起こし修正 ≠ 要約）
- 新規文書作成
- 翻訳作業

## ワークフロー概要

### Phase 0: 計画・分割（必須）

**目的**: Claudeの能力限界に対応

```bash
# ファイルサイズ確認
wc -l target_file.txt

# 分割判断
# 0-300行: 分割不要
# 300-600行: 3段階
# 600-900行: 3-4段階
# 900行以上: 5段階以上
```

**分割しないリスク**:
- ファイル内容を正しく扱えない
- 作業品質が著しく低下
- 修正漏れの増加

### Phase 1: 準備（用語集確認）

**目的**: 修正作業に必要な用語・パターンの把握

**実施内容**:
1. プロジェクト用語集を精読
2. 誤変換パターンを把握
3. 固有名詞（人名・組織名）をリスト化

### Phase 2: メイン作業（修正）

**目的**: 用語集に基づく専門用語の修正、話者特定

**サブフェーズ**:
- 2.1 初期読み込み（会議の流れ把握）
- 2.2 修正作業（カテゴリ別に実施）
- 2.3 検証プロセス（Web検索、参照資料照合）
- 2.4 ユーザー確認（不明瞭箇所のリスト化・質問）

**修正フォーマット例**:
```markdown
## 00:00:00 - 会議開始・挨拶

**話者**: 田中（A社）
> （発言内容）

**話者**: 鈴木（B社）
> （発言内容）
```

### Phase 3: 二次成果物（議事メモ）

**目的**: 修正済み文字起こしから要点を抽出

**議事メモ構成**:
1. 打合せ概要（日時、参加者、目的）
2. 技術要件
3. 提案・検討事項
4. 次回アクション
5. 備考

### Phase 4: レビュー・プッシュ

**目的**: 最終確認とリポジトリ反映

**grep確認例**:
```bash
# 修正が適用されているか確認
grep "正しい用語" modified_file.md | wc -l

# 誤変換が残っていないか確認
grep "誤変換パターン" modified_file.md  # 0件であること
```

## 絶対禁止事項

### 1. ファイル分割の省略

**禁止**: 400行以上のファイルを分割せずに一括処理

### 2. 用語集を確認せずに修正

**禁止**: 推測や一般知識だけで修正

### 3. 文脈無視の一括置換

**禁止**: `sed`等で機械的に一括置換

### 4. 生データファイルの編集

**禁止**: 原本（`_raw.txt`等）の直接編集

### 5. パートの統合

**禁止**: 分割されたパートを1つのファイルに統合

### 6. 要約の作成

**禁止**: 発言内容を省略したサマリファイルの作成

## 品質チェックリスト

### 完全性（Completeness）
- [ ] すべての発言が記録されている
- [ ] 話者が可能な限り明記されている
- [ ] 議題が網羅されている

### 正確性（Accuracy）
- [ ] 用語集に基づいて修正されている
- [ ] 固有名詞が正確
- [ ] 数値が正確
- [ ] 誤変換修正率 95%以上

### 一貫性（Consistency）
- [ ] 表記が統一されている
- [ ] 見出しレベルが適切
- [ ] 話者形式が統一されている

## 完了定義（Definition of Done）

- [ ] Phase 0-4のすべてのチェックリストが完了
- [ ] 用語集に基づく修正がすべて適用されている
- [ ] ユーザー確認プロセスが完了し、回答が記録されている
- [ ] grep確認で修正箇所が確認されている
- [ ] Git pushが完了している

## 汎用化のポイント

このスキルを他のドメインに適用する場合：

1. **用語集の作成**: ドメイン固有の誤変換パターンをリスト化
2. **分割基準の調整**: ファイルサイズに応じた分割戦略
3. **検証パターンの定義**: grep等で確認すべきパターン
4. **二次成果物の定義**: 議事メモ以外（技術レポート等）
