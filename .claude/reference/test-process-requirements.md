# テストプロセス要件

**目的**: スクリプト・コード作成における検証義務と完了基準の明確化
**適用範囲**: technical-projects-cli/を使用する全プロジェクト

---

## 1. 基本原則

### 1.1 完了の定義

**TDD鉄則（obra/superpowers、Phase 4統合）**:
> NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST

これは「まずテストを書き、次にそのテストを通す実装を書く」という原則です。
**詳細**: `docs/rules/test.md` - テスト品質ルール（FP-1対策）

**誤った完了基準（禁止）**:
```
❌ コードを書いた → 完了
❌ ファイルを保存した → 完了
❌ git commitした → 完了
```

**正しい完了基準（必須）**:
```
✅ コードを書いた
✅ テストを実行した
✅ テストが成功した
✅ 実際の動作を確認した
✅ エラーがないことを確認した
→ 完了
```

**式で表すと**:
```
実装 ≠ 完了
実装 + テスト + パス = 完了
```

### 1.2 検証なしの成果物提供は禁止

- 動作確認や検証を行わずに成果物として提供することはありえない
- スクリプト作成と部分/一貫テストの構築および網羅的な実施は義務

## 2. 仕様-テスト対応（トレーサビリティ）

### 2.1 基本構造

```
仕様書の各項目 ←→ 具体的なテスト手段
```

**トレーサビリティマトリクス例**:

| 仕様ID | 仕様項目 | テスト種別 | テスト方法 | 合否基準 |
|-------|---------|----------|----------|---------|
| S001 | ファイル読み込み | 単体 | 入力テスト | エラーなし |
| S002 | データ検証 | 単体 | 異常系テスト | 適切なエラー |
| S003 | 計算処理 | 単体 | 既知入力 | 期待値一致 |
| S004 | 出力生成 | 統合 | 全体フロー | 正しいファイル |

### 2.2 仕様項目ごとのテスト設計

各仕様項目に対して以下を定義:
- 正常系テスト
- 異常系テスト
- 境界値テスト
- 期待結果

## 3. テスト階層

### 3.1 部分テスト（単体テスト）

**対象**: 個々の関数・モジュール

**実施基準**:
```yaml
必須:
  - コア機能の単体テスト
  - 境界値テスト
  - エラーハンドリングテスト
```

### 3.2 一貫テスト（統合テスト）

**対象**: 複数コンポーネントの連携

**実施基準**:
```yaml
必須:
  - クリティカルパスの統合テスト
  - 入力→処理→出力の全体フロー
```

### 3.3 網羅的実施

**カバレッジ目標**:
- プロトタイプ: 主要機能100%
- 本番: コア機能90%以上

## 4. 実装タイプ別テスト要件

### 4.1 Python実装

```yaml
必須テスト:
  1. ユニットテスト:
     - pytest実行
     - 全テストパス確認

  2. 統合テスト:
     - 実際の処理実行
     - 出力ファイル確認

コマンド例:
  - pytest tests/
  - pytest --cov=src tests/
```

### 4.2 JavaScript/HTML実装

```yaml
必須テスト:
  1. ブラウザテスト:
     - HTMLファイルを開く
     - コンソールエラーなし確認

  2. 自動テスト:
     - npm test実行
     - Playwright等で検証

コマンド例:
  - npm test
  - npx playwright test
```

### 4.3 Bashスクリプト実装

```yaml
必須テスト:
  1. 構文チェック:
     - shellcheck実行
     - bash -n でチェック

  2. 動作確認:
     - 実際の環境で実行
     - 期待される出力確認

コマンド例:
  - shellcheck script.sh
  - bash -n script.sh
```

## 5. チェックリスト

### 5.1 実装前

```markdown
- [ ] 仕様項目を列挙したか？
- [ ] 各仕様項目に対するテスト方法を決定したか？
- [ ] テスト環境は整っているか？
```

### 5.2 実装後

```markdown
- [ ] 単体テストを作成したか？
- [ ] 単体テストを実行したか？
- [ ] 単体テストが全てパスしたか？
- [ ] 統合テストを実行したか？
- [ ] エラーログを確認したか？
- [ ] 実際の動作を確認したか？
```

### 5.3 完了宣言前

```markdown
- [ ] 仕様項目すべてにテストが対応しているか？
- [ ] テストパス率 = 100%？
- [ ] 動作確認済み？
- [ ] ユーザーに提供可能な品質？
```

## 6. テスト失敗時の対応

### 6.1 基本原則

```
テスト失敗 = 実装未完了
```

**改ざん防止原則（Phase 4統合）**:
テストを弱めてパスさせるのは禁止。テストを変更せず、実装を修正する。
**詳細**: `docs/rules/test.md` - アンチパターン「テストを弱めてパスさせる」

**禁止行動**:
- ❌ テスト失敗を無視
- ❌ "一部パス"で完了宣言
- ❌ "後で修正"と先送り
- ❌ **テストの期待値を実装に合わせて変更する（改ざん）**
- ❌ **アサーションを削除・コメントアウトする（改ざん）**

**必須行動**:
- ✅ エラーログ確認
- ✅ 原因特定
- ✅ バグ修正（実装側を修正）
- ✅ 再テスト
- ✅ 全テストパス → 完了

### 6.2 デバッグプロトコル

**診断と修正の分離**:
```
診断フェーズ（10分）:
  □ 事実確認（ファイル存在、関数名等）
  □ エラーログ完全解析
  □ 全問題をリストアップ

修正フェーズ（5分）:
  全問題を一度に修正

確認フェーズ:
  コミット・テスト実行（1回）
```

**禁止事項**:
- 推測による修正
- 段階的な試行錯誤
