### 1. ファイル品質管理ルール（CLI版）

#### 必須記載事項
全てのファイル作成時に冒頭への以下の記載を必須化：

```markdown
**ID**: [file_unique_identifier]
**作成日**: YYYY年MM月DD日
**作成者**: [作成者名]
**重要度**: 🚨/⚠️/📝/🔧
**分類**: [スクリプト/レポート/計画/分析/ガイド]
```

#### ID命名規則
- 英小文字・数字・アンダースコア使用
- `[プロジェクト略語]_[内容]_[用途]`の形式
- 例: `universal_instruction_quality_rules_cli`, `project_status_analysis_report`

#### ファイル名とIDの区別（CLI環境）

**重要な区別**:
- **ID（ファイル識別子）**: 文書内の管理用、任意の長さ可
  - 例: `universal_instruction_quality_rules_cli`

- **ファイル名（実ファイル）**: システム制限あり、100文字以内推奨
  - 例: `univ-instr-rules-cli.md`

#### ファイル名正規化の実装（CLI環境）

実際のファイル作成時は、以下の正規化を適用：

**Python実装例**:
```python
import re
import os

def safe_filename_cli(intended_name, max_length=100):
    """CLI環境でファイル名を安全な形式に変換"""
    # パス区切り文字を除去
    name = intended_name.replace('/', '-').replace('\\', '-')

    # 特殊文字を安全な文字に置換
    name = re.sub(r'[^a-zA-Z0-9\-_.]', '-', name)

    # 連続するハイフンを単一に
    name = re.sub(r'-+', '-', name)

    # 拡張子を保護しつつ、本体を制限
    base, ext = os.path.splitext(name)
    if len(base) > (max_length - len(ext) - 1):
        base = base[:(max_length - len(ext) - 1)].rstrip('-')

    return f"{base}{ext}".lower()

# 使用例
original = "Technical-Accuracy-Guidelines-for-Claude-Code-CLI-Comprehensive.md"
safe = safe_filename_cli(original, max_length=100)
# → "technical-accuracy-guidelines-for-claude-code-cli-comprehensive.md" (64文字)
```

#### 命名規則の実践例（CLI環境）

| 意図されたファイル名 | 正規化後（100文字以内） | 評価 |
|------------------|---------------------|------|
| `api-spec.md` | `api-spec.md` | ✅ そのまま使用可 |
| `technical-accuracy-guidelines-comprehensive-version-final.md` | `technical-accuracy-guidelines-comprehensive-version-final.md` | ✅ 使用可（68文字） |
| `My Project/Report v2.txt` | `my-project-report-v2.txt` | ✅ 安全化 |
| `C:\Projects\...\very-long-filename-that-exceeds-limits.md` | `very-long-filename-that-exceeds-limits.md` | ✅ パス除去 |

#### ファイルパスの絶対ルール（CLI環境）

**Claude Code環境**:
```yaml

  推奨: 絶対パス指定
  例: "C:\\Users\\username\\github\\project\\src\\module.py"
  理由: 確実な場所に作成、パス曖昧性の排除

相対パス使用時:
  条件: 作業ディレクトリが明確な場合のみ
  例: "src/module.py"
  注意: 現在のディレクトリを常に意識

ツール使用:
  新規作成: Write(file_path=..., content=...)
  既存編集: Edit(file_path=..., old_string=..., new_string=...)

```

### 2. 重大不始末防止ルール

#### 核心原則
以下の原則をインストラクションに必須追加：

1. **推測判断完全禁止**: 不明点は必ず明確化質問で確認、「思われる」「可能性がある」等の曖昧表現を排除
2. **事前検証必須化**: スクリプト実行、重要決定等は必ず事前検証・段階的確認を実施
3. **正確報告原則**: 確認していない事実を「確認済み」と報告することは重大不始末
4. **直接確認優先**: 最も確実な確認手法を優先選択、複雑な迂回手法を避ける

### 3. 進捗報告による作業継続性の確保

#### 進捗報告フォーマット3種

##### 軽量版（頻繁に使用）
```
[進捗] 完了:○○ / 作業中:△△ / 次:□□
```

##### 標準版（主要な区切りで使用）
```
---進捗---
完了：[簡潔なリスト]
現在：[具体的な作業内容]
次　：[予定されている作業]
---------
```

##### 詳細版（複雑な作業で使用）
```
---作業状況 [識別子]---
《これまで》[主要完了事項]
《現在地》[詳細な現在位置]
《継続情報》[次回開始用の情報]
-----------------------
```

#### 実装の核心原則
- **進捗報告は「作業の自然な一部」として出力**
- **ユーザーは30秒で把握、Claudeは必要な時間で思考**
- **品質を犠牲にした速度追求は本末転倒**

#### 30秒把握の正しい理解

**「30秒把握」とは**：
- ✅ ユーザーが進捗報告を見て30秒で前回の状況を理解できること
- ✅ こまめな進捗報告により、途中停止時の状況が即座に把握可能
- ✅ 次のチャットに進捗報告を貼り付けるだけで作業継続が可能

**「30秒把握」ではないもの**：
- ❌ Claudeの思考時間を30秒に制限すること
- ❌ 再開時の分析や検討を急がせること
- ❌ 品質より速度を優先すること

#### 実装方法
1. **自然な進捗出力**
   - 主要な作業の区切りで進捗を報告
   - 長い応答では2000トークン程度ごとに状況を出力
   - 形式にこだわらず、実用的な情報を提供

2. **30秒把握を可能にする情報設計**
   - 「何を完了し、今何をしており、次は何か」を明確に
   - 技術的詳細より作業の流れを重視
   - コピー＆ペーストで即座に使える形式

3. **思考の自由度**
   - 進捗報告はユーザーの把握を助けるものであり、Claudeの思考時間を制限するものではない
   - 新規問題や複雑な分析には必要な時間をかける

### 4. 段階的検証プロセス

#### 技術作業の基本フロー
1. **計画**: 作業範囲・手順の明確化
2. **検証**: 事前チェック・構文確認
3. **実行**: 段階的実行（テスト→小規模→本番）
4. **確認**: 結果の検証・品質チェック

### 5. 情報収集の信頼性確保

#### 情報源の優先順位設定
1. **最優先**: プロジェクト公式情報・ナレッジベース
2. **高優先**: 公式サイト・一次情報源
3. **中優先**: 信頼性の高い二次情報源
4. **低優先**: 参考程度の情報源
