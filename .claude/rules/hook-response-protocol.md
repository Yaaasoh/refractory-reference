# Hook応答プロトコル

**対策対象**: FP-5（Hook警告無視） | **優先度**: High | **適用**: 全パッケージ

## Hook警告への応答手順（4ステップ）

| Step | 行動 | 内容 |
|------|------|------|
| 1 | **全文読む** | 警告メッセージを最後まで読む |
| 2 | **種類を特定** | テスト改ざん/Lint改ざん/設定変更/破壊的コマンド/セキュリティリスク |
| 3 | **影響を評価** | Critical→即中止＋報告、High/Medium→根本原因特定＋修正、Low→記録＋可能なら改善 |
| 4 | **対応する** | 根本原因を解決し、回避策を使わない |

## 禁止事項

### Hook警告の無視

- 警告を読まずに次の操作を続行する
- 警告の意味を理解せずに無視する
- 「些細な警告」と決めつけて無視する
- Hook警告を繰り返し引き起こす

### Hook機能の無効化

- 警告回避のためにHookを無効化する
- `settings.local.json`にhooks空設定を追加してsettings.jsonのHookを迂回する
- Hookスクリプトをコメントアウトする
- Hookスクリプトに`exit 0`を追加して常に成功させる

### 不適切な対応

- 根本原因を解決せずに回避策を使う
- 「一時的」と称して恒久的に無効化する
- 警告をユーザーに報告しない

## 影響レベル別の対応

| レベル | 対応 |
|--------|------|
| **Critical** | 即座に操作中止 → ユーザーに報告 → 承認後に進行 |
| **High** | 警告確認 → 根本原因特定 → 修正して再試行 |
| **Medium** | 警告確認 → 根本原因特定 → 修正して再試行 |
| **Low** | 情報を記録 → 可能なら改善 |

## 3原則

| 原則 | 内容 |
|------|------|
| **警告優先** | Hook警告対応は元の作業より最優先。後から修正するとコスト増 |
| **根本解決** | Hookを無効化/無視/設定変更するのではなく、テスト修正・実装改善・品質基準維持 |
| **透明性** | Critical/Highレベル・繰り返し発生・ユーザー指示と矛盾する警告は必ずユーザーに報告 |

## 防御層（Multi-layer Defense）

| Layer | 手段 | 効果 | 内容 |
|-------|------|------|------|
| 1 | Rules（本文書） | 弱 | Hook対応プロトコルの提示 |
| 2 | Skills | 中 | 警告解読、対応手順の提示 |
| 3 | Hooks | 強 | 不適切操作の検出・blocked=trueで防止（品質/セキュリティ/改ざん検出） |
