# Plan Mode 計画品質ルール

**対策対象**: 計画フェーズの調査不足、願望リスト型計画
**優先度**: High ｜ **適用**: すべてのパッケージ

## ツール許可/禁止

| 許可 | 禁止 |
|------|------|
| Read, Grep, Glob, Bash（読み取り系） | Edit, Write, NotebookEdit |
| WebSearch, WebFetch, Agent（read-only） | |
| AskUserQuestion | |

## 禁止事項

1. **調査なしの計画作成** — Read/Grep/Globを1回も使わず計画作成、既存コードを読まずに「〜を参考」と記載
2. **願望リスト型計画** — 根拠のない「〜を参考」、未確認ファイル参照、具体性のない箇条書き
3. **計画承認後の再調査** — Normal Modeで既存コードを読み始める、計画にないファイルの調査（発見時はPlan Modeに戻る）
4. **計画承認と実装許可の混同** — 計画承認≠実装許可。「実装してください」の明示指示まで待つ。緊急発見時は報告→判断仰ぐ

## ワークフロー: Explore → Plan → Implement

| フェーズ | モード | 内容 |
|---------|--------|------|
| **Explore** | Plan Mode | Grep/Globで関連ファイル特定→Readで実コード確認→既存パターン把握→AskUserQuestionで要件明確化 |
| **Plan** | Plan Mode | 読んだ事実に基づく設計→「何を・どのファイルで・なぜ」提示→ユーザー承認 |
| **Implement** | Normal Mode | 計画に従い実装→テスト検証→計画外の問題発見時はPlan Modeに戻る |

## Plan Mode使用判断

| 使う | スキップ |
|------|---------|
| 3+ファイル変更、コードベース全体理解必要、方針不確実 | 1-2ファイル明確修正、タイポ修正、1文で説明可能 |

## 計画の最小要件

**「ファイルパス + 変更内容 + 根拠（読んだ事実）」** — 「確認済み」明示、具体的な行番号・件数・フォーマット記載、設計判断に理由を添える

| 詳細度 | 対象 | 計画内容 |
|:------:|------|---------|
| Simple | 1-2ファイル | ファイルパス + 変更概要 |
| Medium | 3-5ファイル | + 既存パターン確認結果 + 設計判断根拠 |
| Complex | 6+ファイル | + 影響範囲分析 + テスト戦略 + リスク項目 |
