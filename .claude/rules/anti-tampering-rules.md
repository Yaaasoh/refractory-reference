# 改ざん防止ルール（汎用版）

**対策対象**: FP-1（テスト改ざん）+ その他改ざん全般 | **優先度**: Critical | **適用**: 全パッケージ

**改ざんの定義**: 問題の本質を解決せず、検証基準を緩めて問題を隠蔽する行為

## 禁止事項

| カテゴリ | 絶対禁止 |
|---------|---------|
| 1. テスト | 期待値を実装バグに合わせる、アサーション削除/コメントアウト、`@skip`/`it.skip`等でスキップ、タイムアウト過度延長 |
| 2. Lint設定 | エラー消去目的のルール無効化、`pylint: disable`/`eslint-disable`の安易な使用、設定ファイルでの全体無効化 |
| 3. CI/CD | ビルド失敗隠蔽のステップコメントアウト、テスト除外、カバレッジ閾値低下、`\|\| true`でエラー無視 |
| 4. 品質基準 | カバレッジ/複雑度/セキュリティ/パフォーマンス基準の緩和 |
| 5. 仕様（INC-018） | 仕様未達時に仕様側を弱める、機能を「将来対応予定」に格下げ、「代替手段で対応」と正当化 |
| 6. 権限境界 | ユーザー設定の権限境界を効率理由で逸脱、許可範囲の無断拡大 |
| 7. 不誠実な対応 | 不備指摘への謝罪なき事務的返答、責任転嫁 |

**原則**: 検査で差異を発見したら、修正対象は常に**実装側**。仕様・テスト・基準を実装に合わせるのは改ざん。

## やるべきこと

### 根本原因の解決（Fix Forward）

> **TDD鉄則**: NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST（obra/superpowers）
> **原則**: 基準を下げるのではなく、実装を上げる

1. 失敗している検証（テスト、lint等）を確認
2. なぜ失敗しているのか原因を特定
3. **実装を修正**して検証をパス
4. 検証基準は維持

### 例外が許される条件

- [ ] Issue/PRで追跡されている
- [ ] スキップ理由が明確に記述されている
- [ ] 修正予定が明示されている
- [ ] ユーザーに報告済み

### 品質維持の原則

カバレッジ・Lintルール・CI/CD・品質基準は**維持または向上のみ**。低下は禁止。

### 不備指摘への対応順序

**謝罪 → 原因説明 → 再発防止**（この順序を厳守）

## 防御層（Multi-layer Defense）

| Layer | 手段 | 効果 | 内容 |
|-------|------|------|------|
| 1 | Rules（本文書） | 弱 | 基本方針の提示 |
| 2 | Skills | 中 | `code-quality-enforcer`, `prompt-purpose-validator` — 改ざんパターン検出・警告 |
| 3 | Hooks | 強 | `quality_check.sh`(PreToolUse), `prevent_false_completion.sh`(UserPromptSubmit) — blocked=trueで停止 |

### Hook検出パターン

- テスト改ざん: `assert`削除、`describe.skip`/`it.skip`/`@skip`
- Lint緩和: `eslint-disable`/`pylint: disable`/`noqa`
- CI改ざん: テストコメントアウト、`|| true`
