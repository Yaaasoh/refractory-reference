# タスク完全性ルール

**対策対象**: FP-6（タスク範囲超過）, FP-7（偽完了報告）
**優先度**: High
**出典**: vibration-diagnosis-prototype失敗事例、WORK_PROCESS_PROTOCOLS Protocol 4

## やってはいけないこと

### 1. タスク範囲の逸脱（FP-6）

**絶対禁止**:
- 指示されていない機能を追加する
- 関連するが別のタスクまで実施する
- リファクタリングを勝手に行う
- 依頼されていないファイルを変更する
- 「ついでに」で追加作業をする

**具体例（vibration-diagnosis-prototype Sin 6）**:
```bash
# タスク: "ログイン機能のバグ修正"

# ❌ 悪い例：タスク範囲を超える
# - ログイン機能の修正（✓ 指示通り）
# - ついでにログアウト機能も実装（✗ 指示外）
# - ついでに認証ミドルウェアもリファクタリング（✗ 指示外）
# - ついでにパスワードポリシーも強化（✗ 指示外）

# ✅ 正しい例：タスク範囲を守る
# - ログイン機能の修正のみ
# - その他の改善提案はユーザーに報告（実施は承認後）
```

### 2. 偽完了報告（FP-7）

**禁止**:
- タスクが未完了なのに「完了」と報告
- 一部だけ完了して「完了」と報告
- エラーが残っているのに「ほぼ完了」と報告
- テスト未実施で「完了」と報告
- 確認不足で「完了」と報告

**具体例（vibration-diagnosis-prototype Sin 7）**:
```markdown
# ❌ 悪い例：偽完了報告
「ログイン機能の修正が完了しました」
→ 実際: テストが1つ失敗している

# ✅ 正しい例：正確な状況報告
「ログイン機能の修正を実施しましたが、test_login_with_invalid_passwordが失敗しています。
原因を調査して修正します」
```

### 3. 無断の追加作業

**禁止**:
- ユーザーに確認せずに追加機能を実装
- 「良かれと思って」で勝手に変更
- 「ベストプラクティスだから」で勝手に導入
- リファクタリングを無断で実施

## やるべきこと

### 1. タスク範囲の明確化

**タスク開始前のチェック**:
- [ ] タスクの目的を理解したか？
- [ ] タスクの範囲（スコープ）を理解したか？
- [ ] 不明点はユーザーに質問したか？（AskUserQuestion）
- [ ] 期待される成果物を確認したか？

**スコープ確認の例**:
```markdown
# ユーザーの指示: "ログイン機能のバグを修正して"

## スコープ確認（AskUserQuestion）
**質問**: ログイン機能の修正について、以下を確認させてください

**確認事項**:
1. 修正対象: login関数のみでよろしいですか？
2. テスト: 既存のテストがパスすれば完了でよろしいですか？
3. その他: ログアウト機能やセッション管理は対象外でよろしいですか？

**推奨**:
- 修正範囲: login関数のみ
- テスト: 既存テスト + 新規テスト（バグ再発防止）
- その他: 関連する改善提案があれば報告（実施は別タスク）
```

### 2. TodoWriteでタスク管理

**タスク分割の例**:
```javascript
// タスク: "ユーザー認証機能の改善"

// TodoWrite で明確化
[
  {
    content: "ログイン機能のバグ修正",
    status: "in_progress",
    activeForm: "Fixing login bug"
  },
  {
    content: "ログイン機能のテスト追加",
    status: "pending",
    activeForm: "Adding login tests"
  },
  {
    content: "修正内容のドキュメント更新",
    status: "pending",
    activeForm: "Updating documentation"
  }
]

// ✅ 各タスクを順番に完了、完了報告は全タスク完了後
```

### 3. 完了基準の確認（Definition of Done）

**技術的完了基準**:
- [ ] 指示された機能・修正が完了
- [ ] すべてのテストがパス
- [ ] コードレビュー基準を満たす
- [ ] ドキュメント更新（必要な場合）

**報告基準**:
- [ ] すべての技術的完了基準を満たしている
- [ ] エラー・警告がない
- [ ] ユーザーの期待に沿っている

**完了報告の例**:
```markdown
✅ タスク完了: ログイン機能のバグ修正

**実施内容**:
- [ src/auth/login.ts:45 ] パスワード検証ロジックの修正
- [ tests/auth/login.test.ts:120 ] テストケース追加（バグ再発防止）

**確認済み項目**:
- [x] すべてのテストがパス（27 passed, 0 failed）
- [x] 既存機能に影響なし
- [x] コードレビュー基準を満たす

**変更ファイル**:
- src/auth/login.ts
- tests/auth/login.test.ts

**その他**:
修正中に以下の改善機会を発見しました（実施は別途ご相談）:
- ログアウト機能のセッションクリア処理の改善
- パスワードポリシーの強化（現在: 6文字以上 → 推奨: 8文字以上）
```

### 4. 追加提案の報告方法

**発見した改善機会の報告**:
```markdown
# タスク完了報告と分離して報告

✅ ログイン機能のバグ修正が完了しました

---

💡 改善提案（別タスクとして検討いただけますか？）

**提案1: ログアウト機能の改善**
- 現状: セッションが完全にクリアされない場合がある
- 提案: セッションクリア処理を強化
- 優先度: 中
- 工数見積: 1-2時間

**提案2: パスワードポリシー強化**
- 現状: 6文字以上
- 提案: 8文字以上 + 英数字混在
- 優先度: 低
- 工数見積: 2-3時間
```

## 例外処理

### 例外が許される場合

1. **セキュリティ脆弱性の発見**
   - **条件**: タスク実施中に重大な脆弱性を発見
   - **手順**:
     1. ユーザーに即座に報告
     2. 承認を得てから修正
     3. 修正内容を明記

2. **破壊的バグの発見**
   - **条件**: データ損失・システムダウンに繋がるバグを発見
   - **手順**:
     1. ユーザーに緊急報告
     2. 承認を得てから修正
     3. 元のタスクと分離して報告

**報告例**:
```markdown
⚠️ 緊急報告: セキュリティ脆弱性を発見

**元のタスク**: ログイン機能のバグ修正

**発見した問題**:
src/auth/session.ts:67 でセッショントークンが平文でログ出力されています。
これは情報漏洩のリスクがあります。

**提案**:
元のタスクとは別に、この脆弱性を修正してもよろしいでしょうか？
```

## 防御層（Multi-layer Defense）

### Layer 1: Rules（本ドキュメント）
- **効果**: 弱（LLMが無視する可能性あり）
- **役割**: 基本方針の提示

### Layer 2: Skills
- **効果**: 中（タスク開始時に起動）
- **スキル**: `verification-enforcer`
- **機能**:
  - タスク範囲の明確化
  - 完了基準の確認
  - 完了報告前のチェックリスト提示

### Layer 3: Hooks
- **効果**: 強（完了報告をブロック）
- **フック**: `prevent_false_completion.sh`（UserPromptSubmit）
- **機能**:
  - 完了報告の検出（「完了」「done」「finished」等）
  - 未完了タスクの存在チェック
  - テスト失敗の検出
  - エラー・警告の検出

**Hookの例**:
```bash
#!/bin/bash
# .claude/hooks/prevent_false_completion.sh

INPUT=$(cat)
PROMPT=$(echo "$INPUT" | jq -r '.user_prompt // empty')

# 完了報告を検出
if echo "$PROMPT" | grep -iE "(完了|done|finished|completed)" > /dev/null; then
  # テスト失敗をチェック
  if git diff --cached | grep -E "(\-.*assert|describe\.skip|it\.skip)" > /dev/null; then
    echo "⚠️  警告: テストの削除・スキップを検出しました" >&2
    echo "偽完了報告の可能性があります" >&2
    echo "" >&2
    echo '{"blocked": true, "reason": "Test modification detected"}' | jq
    exit 0
  fi
fi

echo "$INPUT"
```

## ベストプラクティス

### 1. Single Responsibility（単一責任）

**原則**:
> 1つのタスクは1つの目的のみを達成する

**実践**:
```markdown
# ❌ 悪い例：複数の目的が混在
タスク: "ログイン機能の改善"
- バグ修正
- パフォーマンス改善
- UI改善
- セキュリティ強化

# ✅ 正しい例：1タスク1目的
タスク1: "ログイン機能のバグ修正"
タスク2: "ログイン機能のパフォーマンス改善"
タスク3: "ログイン画面のUI改善"
タスク4: "ログイン機能のセキュリティ強化"
```

### 2. Scope Creep Prevention（スコープ拡大の防止）

**スコープ拡大のサイン**:
- 「ついでに」と考え始めた
- 元のタスクと関連するが別の問題を見つけた
- 「これも直した方がいい」と思った

**対処法**:
1. 一旦停止
2. ユーザーに報告・相談
3. 承認を得てから実施

### 3. Honest Reporting（正直な報告）

**報告の原則**:
- 完了は完了、未完了は未完了
- 「ほぼ」「だいたい」は使わない
- 残課題・制約事項を明記
- できたこと・できなかったことを明確に分ける

**報告例**:
```markdown
# ✅ 正直な報告
## 完了した項目
- ログイン機能のバグ修正
- バグ再発防止のテスト追加

## 未完了の項目
- パスワードリセット機能のテスト（test_password_reset が失敗）
  → 原因調査中、次のセッションで対応予定

## 制約事項
- IE11では動作未確認（環境がないため）
```
