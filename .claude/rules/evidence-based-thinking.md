# 証拠ベース思考ルール

**対策対象**: FP-8（推測による実装）
**適用範囲**: すべてのパッケージ

## 定義

| 証拠（使うべき） | 推測（禁止） |
|-----------------|-------------|
| 実際に確認したドキュメント | 「〜だろう」に基づく判断 |
| 実際に読んだコード | コードを読まず判断 |
| 実際に実行した結果 | テストせず「動くはず」 |
| 実際に受け取ったエラーメッセージ | エラーを読まず「よくあるエラー」と決めつけ |

## 禁止事項

1. **推測で実装** — コード/ドキュメント/エラーを読まずに作業を進めること
2. **推測を事実として述べる** — 「〜が原因です」は確認した場合のみ。推測なら「推測ですが」と明記
3. **責任転嫁** — 外部要因のせいにする前に自分の操作を疑う
4. **WebSearch結果を捨てる** — `WebSearch/WebFetch実行回数 = 保存ファイル数`
5. **コマンド/オプションの存在を推測で提示** — `--help`で確認してから提示

## 3ステップアプローチ: Read → Verify → Implement

### Step 1: Read（読む）
- **コード**: `grep -A 20 "def func_name"` / Readツールで関数定義・型ヒント・使用例を確認
- **ドキュメント**: README.md → CLAUDE.md → API仕様 → ARCHITECTURE.md の順
- **コマンド**: `where <command>` で存在確認、`<command> --help` でオプション確認

### Step 2: Verify（確認する）
- 実際に動かして確認（`python -c "..."` / `curl` / テスト実行）
- エラーメッセージは**全文**読む（一部だけ見て判断しない）
- スタックトレースを追跡して問題箇所を特定

### Step 3: Implement（実装する）
- 確認した証拠に基づいて実装
- 証拠をコミットメッセージやコメントに記録

## チェックリスト

**作業前**: 関連コードを読んだか？ ドキュメントを読んだか？ 既存の使用例を確認したか？
**作業中**: 推測ではなく確認した事実に基づいているか？ エラー全文を読んだか？
**作業後**: 動作を確認したか？ テストで検証したか？

## Skill使用時: 具体→抽象の順序

skill使用前に **Phase 0** を実施:
1. 過去の成果物を最低1件確認（具体的パターン把握）
2. ファイル構成・命名規則・分割基準を把握
3. skillドキュメントを確認（抽象）
4. 把握したパターンに基づいて計画

**禁止**: skillドキュメント（抽象）のみ確認して作業開始

## 原因不明時の対応

1. エラーメッセージを全文読む
2. 関連ツールの状態を確認する（`lms status` / `nvidia-smi` / バージョン確認等）
3. **確認した事実のみを報告する**
4. 推測が必要な場合は「推測ですが」と明記する

## 防御層

| Layer | 手段 | 効果 |
|-------|------|------|
| Rules | 本ドキュメント | 弱（LLM無視可能性あり） |
| Skills | root-cause-analyzer等 | 中（証拠収集誘導） |
| Hooks | 証拠ベースはHook強制困難 | Skill自動起動で補完 |
