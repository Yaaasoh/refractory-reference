# 文書品質ルール

**対策対象**: 文書構造崩壊、自己申告による検証の形骸化
**優先度**: Medium ｜ **適用**: すべてのパッケージ

## 核心原則

**機械検証 > 自己申告** — LLMの「問題ありません」は検証にならない

## 禁止事項

1. **自己申告による検証** — 「問題ありません」と自己判断、スクリプト未実行で「目視確認しました」、制約未確認で完了報告
2. **スキーマなしの構造管理** — 繰り返し作成する文書にスキーマ未定義、暗黙ルールの口頭伝達のみ運用
3. **生成者による自己検証** — 文書生成エージェントが自分で検証してOK判断

## 5原則

### 原則1: 検証の優先順位

1. スクリプトによる自動検証（最も信頼性高）
2. サブエージェント（Taskツール）による検証（生成者と別コンテキスト）
3. 人間による確認（最終判断）

### 原則2: スキーマ駆動

繰り返し作成する文書にはスキーマYAMLを定義（配置先: `shared/docs/templates/schemas/`）:

```yaml
metadata:
  format_name: "作業報告書"
sections:
  - id: overview
    title: "概要"
    required: true
    constraints: { min_chars: 50, max_chars: 500 }
```

### 原則3: 生成者と検証者の分離

メインエージェント生成 → PostToolUseフックでL1検証 → サブエージェントでL2-L3検証

### 原則4: 制約の明示

| 制約タイプ | 用途 | 例 |
|-----------|------|-----|
| `min_chars` / `max_chars` | 文字数制限 | 概要: 50〜500文字 |
| `min_items` / `max_items` | 項目数制限 | 次のステップ: 1〜5件 |

### 原則5: 段階的検証（L1→L2→L3）

| レベル | 検証内容 | 実行方法 | 適用場面 |
|:------:|---------|---------|---------|
| L1 構造 | セクション存在・順序・制約 | スクリプト自動実行 | 全文書（作業報告書等） |
| L2 内容 | データ一致・固有名詞・禁止パターン | 手動/サブエージェント | 調査報告書 |
| L3 セマンティック | 忠実度・捏造検出・欠落検出 | サブエージェント | 外部提出文書 |

## 防御層

| Layer | 手段 | 効果 |
|:-----:|------|:----:|
| L1 Rules | 本文書（品質基準提示） | 弱 |
| L2 Skills | `format-writer`（スキーマ→生成→検証統合） | 中 |
| L3 Hooks | `auto-validate.sh`（Write/Edit時に構造検証自動実行） | 強 |
