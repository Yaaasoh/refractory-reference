# デプロイ・検証ルール

**対策対象**: FP-9（デプロイ検証不足） | **優先度**: High | **出典**: vibration-diagnosis-prototype失敗事例

> **核心原則**: デプロイ ≠ 完了 = 検証済み

## 禁止事項

| カテゴリ | 絶対禁止 |
|---------|---------|
| 検証不足 | デプロイ後に結果未確認で完了報告、エラー無視して「完了」、ログ未確認で次タスクへ |
| 中間報告 | ビルド成功のみで「完了」、デプロイ成功だが動作未確認で「完了」、CI実行中に「完了」 |
| エラー隠蔽 | エラーログ無視、警告を「些細」として無視、失敗を「一時的」として放置 |
| 配置≠有効化 | hookファイルコピーのみで「設定完了」、settings.json登録未確認、動作確認なしで「導入済み」 |
| deploy.sh -f | ユーザー明示指示なしでの-f使用、新規ファイル追加に-f使用、対象を根拠なく断定 |

**-f が危険な理由**: リポジトリ固有カスタマイズを全て上書き破壊する

## デプロイ前チェック

- [ ] テスト全パス
- [ ] ビルド成功
- [ ] Lintエラーなし
- [ ] 型チェック成功
- [ ] 依存関係確認済み

## デプロイ中の監視

- [ ] スクリプト出力を読む
- [ ] エラー/警告メッセージの有無確認
- [ ] 進捗確認
- [ ] タイムアウトしていないか確認

## デプロイ後検証（必須）

- [ ] ヘルスチェック成功
- [ ] 主要機能の動作確認
- [ ] エラーログなし（または許容範囲内）
- [ ] パフォーマンス低下なし

## 完了の定義（Definition of Done）

- [ ] デプロイスクリプトが正常終了（exit code 0）
- [ ] ヘルスチェック成功
- [ ] 主要機能が動作確認済み
- [ ] エラーログなし
- [ ] パフォーマンス低下なし
- [ ] ロールバック手順を確認済み

## ファイル配置の完了基準

1. ファイルを配置する
2. settings.json等で**有効化**されていることを確認する
3. 実際に**1回実行**して動作確認する
4. 確認結果をもって完了報告する

## 例外

| 場面 | 条件 |
|------|------|
| Staging環境 | 最小確認（ヘルスチェックのみ可）、「Staging完了」と明記 |
| ホットフィックス | 最小修正→迅速デプロイ→**事後に詳細検証** |

## 防御層（Multi-layer Defense）

| Layer | 手段 | 効果 | 内容 |
|-------|------|------|------|
| 1 | Rules（本文書） | 弱 | 基本方針の提示 |
| 2 | Skills | 中 | `deployment-verifier` — デプロイ後検証誘導 |
| 3 | Hooks | 強 | `post_deploy_verification.sh`(PostToolUse:Bash) — deploy/publish/release検出→検証強制 |
