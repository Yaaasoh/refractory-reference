# 文字起こし作業ルール

**対策対象**: 文字起こし修正作業における各種問題
**優先度**: High ｜ **出典**: INC-013（2026-01-07）

## 5つの絶対禁止事項

1. **Phase 0スキップ** — 過去成果物（生データ・修正版・分割パターン・命名規則・進捗報告書）を確認せず作業開始は禁止
2. **生データファイルの編集** — `_raw.txt`等の直接編集禁止。修正版は`.md`として別ファイルに新規作成
3. **パートの統合** — 分割されたパートを1ファイルに統合禁止（参照性・更新性低下）
4. **要約の作成** — 発言内容を省略したサマリ作成禁止。文字起こし修正≠要約。誤変換部分のみ修正
5. **「構造化」という用語の使用** — INC-013で思考停止の原因。代替:「見出し追加」「話者明示」等の具体的表現

## 検証プロセス（必須）

1. 1行ずつ読み、問題を特定（誤変換/聞き取りミス/話者混同）
2. 用語集・文脈を参照して修正→修正結果を検証
3. 不明点・不自然な点をメモ
4. 全行完了後、再精査
5. 疑問点をユーザーに質問
6. ユーザー承認

## 完了報告の要件

| 必須項目 | 例 |
|---------|-----|
| 修正箇所数 | 誤変換修正: 42箇所、話者特定: 15箇所 |
| 具体的な修正例 | 「滝田さん」→「JAXAさん」 |
| 要確認事項 | 3箇所の専門用語について確認が必要 |

**禁止**: 「確認しました」だけの完了報告、検証なしの完了宣言、数値・例なしの報告

## Definition of Done

- [ ] Phase 0完了（過去成果物確認）
- [ ] 1行1行検証した
- [ ] 用語集パターンをすべて確認した
- [ ] 不明点をユーザーに質問した
- [ ] 具体的な修正結果を報告できる

## 防御層

| Layer | 手段 | 効果 |
|:-----:|------|:----:|
| L1 Rules | 本文書（基本方針提示） | 弱 |
| L2 Skills | `transcript-correction`（作業手順誘導・品質チェック） | 中 |
| L3 Hooks | `transcript_work_check.sh`（生データ編集ブロック）、`prevent_false_completion.sh`（偽完了検出） | 強 |
