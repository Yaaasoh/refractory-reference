# 作業完了フロー

## 対象テーマ

$ARGUMENTS

---

## 設計原則

**2段構成**: 全ての調査・判断を先に完了し（Stage 1）、その後に書き込み・操作を実行する（Stage 2）。

**順序の根拠**: 報告書→タスク検討→ドキュメント更新→プッシュ の順序は依存関係に基づく:
- ドキュメント更新（MASTER_PLAN「次の優先事項」等）はタスク検討結果を必要とする
- タスク検討は報告書（今セッションの成果・残作業）を必要とする

**Plan Mode互換**: Stage 1はread-only操作のみ。Plan Mode中でも動作する。
Stage 1完了後、Plan Mode中ならExitPlanModeを呼んでからStage 2に進む。

---

## Stage 1: 調査・決定（read-only）

全情報を収集し、全決定を下す。このStageではファイルを書かない。

### 1.1 作業状態の把握

```bash
git status
git diff --stat
git log --oneline -10
find . -name "*REPORT*.md" | head -5
```

- 未コミット変更、未プッシュコミット、untracked filesを確認
- 既存報告書の保存場所・命名規則を確認

### 1.2 報告内容の決定

コミットログ・diff・作業記憶から、報告書に書くべき内容を整理する:

- 概要（1-2文）
- 実施内容（コミットハッシュ付き）
- 成果物一覧
- 課題・残作業（タスクID、ブロッカー有無、対象ファイル）
- 保存先パスの決定

### 1.3 タスク検討・引継ぎ設計

**Agentツール（Plan）で次ステップを検討する。**

まず参照ドキュメントの実パスを取得:

```bash
ls MASTER_PLAN.md 2>/dev/null
ls work/HANDOFF_*.md 2>/dev/null | tail -1
ls incidents/*.md 2>/dev/null | tail -5
```

Plan agentに渡す:

```
Agent(subagent_type="Plan", description="Plan next steps"):
  本セッションの作業結果と以下のドキュメントを踏まえて、
  今後のステップをじっくり検討してください。
  「じっくり」= 全参照ドキュメントを実際に読み、網羅的に検討すること。

  参照ドキュメント（すべて読むこと）:
  1. [MASTER_PLAN.mdのパス] — 全体進捗・次の優先事項
  2. [直近HANDOFF_*.mdのパス] — 前回からの引継ぎタスク
  3. 本セッションの成果と残作業（1.2の内容を要約して渡す）
  4. incidents/ 配下 — 未解決のインシデント

  検討観点:
  - 本セッションの成果が他のタスクを unblock したか
  - 前回HANDOFFの残タスクの消化状況（完了/未着手/部分完了）
  - 依存関係の整理（何が何の前提条件か）
  - 同一対象に複数タスクがある場合のグルーピング判断
  - リスクの高いタスクの特定
  - 見落としている作業はないか

  出力形式:
  - 優先度付きタスクリスト（Tier 1/2/3）
  - 各タスクの具体的なサブタスク（番号付き手順）
  - 各タスクの前提条件と依存関係
  - 参照すべきファイルパス
  - 推定規模（行数、ファイル数）
```

**Tier分類の判断基準**:

| Tier | 基準 | 例 |
|------|------|-----|
| Tier 1 | 他タスクのブロッカー、または緊急修正 | バグ修正、前提タスク |
| Tier 2 | 独立実施可能、中程度の重要度 | 機能追加、整備 |
| Tier 3 | 長期・低優先、急がない | リファクタリング、調査 |

**同一対象に複数優先度のタスクがある場合**:

まとめて1セッションで着手すべきか、優先度通り個別に扱うべきかの判断根拠を記載する。

### 1.4 ドキュメント更新判定

1.2（報告内容）と1.3（タスク検討結果）の**両方**を踏まえて、各ドキュメントの更新要否と内容を決定する。

**Agentツール（Explore）で更新対象を確認する:**

```
Agent(subagent_type="Explore", description="Check docs needing update"):
  以下のドキュメントを確認し、本セッションの作業内容に照らして
  更新が必要かどうかを判定してください。

  確認対象:
  1. CLAUDE.md — 「直近の学習メモ」セクション。新たな教訓・知見があるか
  2. MASTER_PLAN.md — 進捗サマリーと「次の優先事項」の両方
  3. README.md — リポジトリ概要に影響する変更があるか
  4. 関連docs/*.md — 作業内容と矛盾する既存記載がないか

  作業内容の要約: [1.2の内容]
  タスク検討結果の要約: [1.3の内容]

  各ドキュメントについて以下を報告:
  - 更新必要 / 不要
  - 必要な場合: 更新すべき箇所と内容
```

**CLAUDE.md 学習メモの判定**:

| 状態 | 対応 |
|------|------|
| 新知見あり、5件以内 | 追記 |
| 新知見あり、5件超 | 古い項目をLEARNING_NOTES.mdに移動してから追記 |
| 新知見なし | スキップ |

### 1.5 移行判定（Stage 2移行チェック）

以下が全て「はい」ならStage 2へ進む:

- [ ] 報告書の内容と保存先が決まっているか
- [ ] タスク検討が完了し、Tier別リストが整理されているか
- [ ] HANDOFF文書の内容が決まっているか
- [ ] ドキュメント更新箇所と内容が決まっているか
- [ ] コミット対象ファイルが確定しているか

**Plan Mode中の場合**: ここでExitPlanModeを呼び、Normal Modeに戻ってからStage 2を実行する。

## Stage 2: 実行（write）

Stage 1で決定した内容に基づき、機械的にファイル作成・更新・コミットを実行する。

### 2.1 報告書の作成

Stage 1.2で決定した内容をWriteする。

**ファイル名**: `WORK_REPORT_YYYYMMDD_[テーマ].md`
**詳細な手順**: `/report` スキルを参照

**報告書テンプレート**:

```markdown
# 作業報告書: [タイトル]

**日付**: YYYY-MM-DD
**作業者**: Claude [モデル名]

---

## 概要

[作業の目的と結果を1-2文で]

## 実施内容

### 1. [作業項目1]（`コミットハッシュ`）
- 実施内容の詳細

## 成果物

| ファイル | 内容 |
|----------|------|
| path/to/file | 説明 |

## 課題・残作業

- T1: [概要] — [ブロッカー有無]。対象: [ファイル/システム]

## 次のステップ

1. [次に行うべきこと]
```

### 2.2 HANDOFF文書の作成

Stage 1.3のタスク検討結果をWriteする。

**保存先**: `work/HANDOFF_YYYYMMDD_NEXT_STEPS.md`

```markdown
# 引継ぎ連絡文 — YYYY-MM-DD セッション終了時点

**作成日**: YYYY-MM-DD
**前回コミット**: `[git log --oneline -1 の結果]`

---

## 今セッションで完了した作業

1. **[タスク名]** (`[コミットハッシュ]`)
   - [実施内容の要約]

---

## 未完了タスク一覧（優先度順）

### Tier 1: 最高優先度

#### T[N]: [タスク名]（[対象システム/ファイル]）

**サブタスク**:
1. [具体的なアクション]
   - 期待結果: [何が得られるか]
   - 検証: [確認コマンドまたは手順]

**前提条件**: [依存タスク、外部条件]
**参照**: [関連ファイルパス]

### Tier 2: 中優先度
...

### Tier 3: 低優先度・長期
...

---

## 次のセッション向けプロンプト

### プロンプト1: [タスク名]

各プロンプトに以下を含めること:
- **背景**: なぜこのタスクが必要か。前セッションで何が完了したか
- **参照ドキュメント**: 読むべきファイルのパス（概要付き、5件以内）
- **タスク（サブタスク付き）**: 番号付き手順。各サブタスクに期待結果と検証手順
- **完了条件**: テストパス、ドキュメント更新、デプロイ確認等

---

## ファイル所在まとめ

| ファイル | 役割 |
|---------|------|
| ... | ... |
```

**注意**: 引継ぎプロンプトは、次セッションのClaude Codeが前提知識なしで作業開始できる自己完結的な内容にすること。

### 2.3 ドキュメント更新

Stage 1.4の判定結果に基づき、「更新要」と判定されたドキュメントのみEditする。

**詳細な手順**: `/update-docs` スキルを参照

MASTER_PLANの「次の優先事項」も含め、全ドキュメントを一度に更新する（タスク検討が完了済みのため、保留する必要がない）。

### 2.4 コミット＆プッシュ

**詳細な手順**: `/safe-push` スキルを参照

```bash
# 1. 現状確認
git status
git diff

# 2. リモート状態確認
git fetch --all
git status

# 3. ステージング（ファイルを明示指定）
git add [対象ファイル]

# 4. コミット
git commit -m "$(cat <<'EOF'
[コミットメッセージ]

Co-Authored-By: Claude Opus 4.6 (1M context) <noreply@anthropic.com>
EOF
)"

# 5. プッシュ
git push origin [ブランチ名]

# 6. 結果確認
git status
git log --oneline -1
```

チェックリスト:
- [ ] 変更ファイルは意図したものか
- [ ] HANDOFF文書が含まれているか
- [ ] 機密情報が含まれていないか
- [ ] `.gitignore` 対象が含まれていないか
- [ ] リモートより behind の場合は先に pull

## セルフチェック

Stage 2完了後、以下を確認:

- [ ] 全Tier 1タスクに3つ以上のサブタスクがあるか
- [ ] 全プロンプトに「背景・参照・タスク・完了条件」があるか
- [ ] 「前回の続き」等の曖昧表現がないか
- [ ] ファイル所在まとめが含まれているか
- [ ] 完了した作業にコミットハッシュが付いているか
- [ ] 同一対象の複数タスクにグルーピング判断があるか

---

## 注意事項

- Stage 1→2の順序を守る（書き始める前に全決定を完了する）
- 問題発生時は該当ステップで停止し報告
- `git push --force` は禁止
- 保存先・更新箇所が不明な場合はユーザー確認

## 関連コマンド

- `/report` - 報告書のみ作成
- `/update-docs` - ドキュメント更新のみ
- `/safe-push` - プッシュのみ
