# コミュニケーション原則

**作成日**: 2026-01-06
**目的**: ユーザーとのコミュニケーションにおける絶対的原則
**重要度**: CRITICAL

---

## 第一原則: ファイル参照の絶対禁止

### 禁止表現

```
❌ 「XXXファイルを参照してください」
❌ 「XXXに記載されています」
❌ 「XXXをコピー&ペーストしてください」
❌ 「XXXをご覧ください」
```

### 正しい表現

```
✅ 「以下が内容です：（全文表示）」
✅ 「ファイルパス: /full/path/to/file

内容:
（全文表示）」
```

---

## 原則の背景

### ユーザーが見ることができないもの

1. Claude Codeの内部ファイルシステム
2. 仮想環境内のファイル
3. 作業ディレクトリの内容（明示的に共有しない限り）

### なぜ参照が問題なのか

1. ユーザーはファイルを直接開けない
2. パスだけ提示しても意味がない
3. ユーザーに不必要な手間をかける
4. 情報が伝わらない

---

## 正しい情報提示方法

### パターン1: 内容を直接表示

```markdown
以下がPR説明文です：

## 概要
機能Xを実装しました。

## 変更内容
- ファイルAを追加
- ファイルBを修正

（全文を表示）
```

### パターン2: パス + 内容表示

```markdown
ファイルパス: `/path/to/file.md`

内容:
---
# タイトル

本文...
---
```

### パターン3: コードブロックで表示

```markdown
作成したスクリプト（`script.sh`）:

\`\`\`bash
#!/bin/bash
echo "Hello"
\`\`\`
```

---

## 情報提示のチェックリスト

### 提示前確認
- [ ] 「参照してください」という表現を使っていない
- [ ] ファイル内容を直接表示している
- [ ] パスを明記している（必要な場合）

### 提示内容確認
- [ ] ユーザーが追加の操作なしで内容を理解できる
- [ ] 必要な情報がすべて含まれている
- [ ] コードはコードブロックで囲んでいる
